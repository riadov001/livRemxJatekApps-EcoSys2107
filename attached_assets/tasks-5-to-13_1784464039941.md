### What & Why

Remplacer les 8 restaurants fictifs actuels par 10 établissements réellement existants à Oujda, avec leurs vraies adresses, spécialités, et des menus réalistes. SmashBurger doit être lié au compte merchant `smashburger.owner@jatek.app` (user id=119). Les autres restaurants restent sous l'admin (owner_id=1). L'objectif est que le contenu de la plateforme reflète la réalité de la ville pour les tests et les démonstrations.

### Done looks like

- Les 8 restaurants fictifs actuels sont remplacés (ou les nouveaux sont insérés et les fictifs supprimés)
- 10 restaurants réels d'Oujda présents en base, chacun avec une spécialité différente
- SmashBurger est lié à owner_id=119 (smashburger.owner@jatek.app)

Vérifier les adresses sur Google Maps ou sources locales.
2. **Suppression des faux restaurants** — Supprimer les 8 restaurants fictifs actuels (id 1–8, owner_id=1) ainsi que leurs menu items associés, en cascade ou manuellement dans le bon ordre (menu_items d'abord, puis restaurants).

## Task #13 — Monitoring heap correct + panneau App Config mobile

**Statut :** ❌ Archived  
**Dépend de :** —

### What & Why

Deux améliorations dans l'espace Système du dashboard :

1. **Monitoring** : la barre Heap Node.js doit afficher le max réel (4096 MB depuis `--max-old-space-size=4096`) et montrer la consommation actuelle en pourcentage correct. Actuellement l'API ne retourne pas le max configuré donc le % est faux.
2. **App Config** : les clés de configuration que l'app mobile lit (langue par défaut, contenu featured, config carousel) doivent être visibles et modifiables depuis le dashboard admin via une nouvelle page `/app-config`, contrôlée via l'API.

### Done looks like

**Monitoring — Heap**
- La card "Heap Node.js" affiche `heapUsed / 4096 MB` et la barre reflète ce ratio
- Exemple : heapUsed ≈ 400 MB → barre ≈ 10% · heapUsed ≈ 2 GB → barre ≈ 50%
- Le max configuré 4096 MB est transmis par l'API dans la réponse `/api/backend/system`

**Monitoring — App Storage**
- Nouvelle section "Stockage App Mobile" listant les clés AsyncStorage connues (`jatek_cart_v4`, `jatek_lang_v1`, `jatek_favorites_v1`, etc.) avec leur description et valeur par défaut contrôlable depuis le serveur

**Page /app-config**
- Nouvelle page admin permettant de modifier :
  - Langue par défaut (`fr` / `ar`)
  - Mode maintenance ON/OFF (synchronisé avec les paramètres plateforme)
  - Nombre de restaurants "featured" affichés en home
  - Ordre des sections home (bannières, catégories, featured, tous)
  - Message d'accueil / slogan affiché dans l'app
- Endpoint `/api/app-config` (GET public, sans auth) retourne cette config — l'app mobile peut le consommer au démarrage

### Out of scope

- Lire/écrire directement le AsyncStorage des téléphones des utilisateurs (impossible côté serveur)
- Historique des modifications de config
- Config AB testing

### Steps

1. **API `/api/backend/system` — exposer le max heap** — inclure `heapMaxConfigured: 4096` (MB) dans la réponse JSON
2. **Monitoring page — corriger la barre heap** — utiliser `heapMaxConfigured` au lieu de `heapTotal` ; afficher `X MB / 4096 MB` dans le label
3. **DB — table `app_config`** — créer une table `app_config` (key VARCHAR PK, value JSONB, updatedAt) et pusher le schéma
4. **API publique `/api/app-config`** (GET, no auth) — retourner un objet JSON plat avec toutes les clés et leurs valeurs (fallback sur les défauts si table vide)
5. **API admin `/api/backend/app-config`** (GET + PUT, auth admin) — lire et écrire les clés de config
6. **Page `/app-config` dans le dashboard** — contrôles : Switch langue/maintenance, NumberInput featured count, liste réordonnée sections home, Input slogan ; sauvegarde via PUT
7. **Section App Storage dans Monitoring** — card listant les clés AsyncStorage connues avec description et valeur par défaut configurée côté serveur

### Relevant files

- `artifacts/backend-dashboard/src/pages/monitoring.tsx`
- `artifacts/api-server/src/routes/backend.ts`
